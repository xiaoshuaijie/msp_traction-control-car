# 按键驱动 NRF24L01 周期发送设计

## 目标

在 `src/app_main.cpp` 中保留 K1-K4 当前的驾驶控制行为，同时让最后一次按下的
按键选择 NRF24L01 周期发送载荷：K1-K4 分别令
`nrf24l01_tx_request.payload[0]` 为 `1`、`2`、`3`、`4`。

上电后未按键时不发送。第一次按键后尽快发送一次，此后每隔 1000 ms 持续发送
当前值，直到其他按键切换发送值。

## 状态与数据流

`app_main()` 增加两个局部状态：

- 发送启用状态：区分尚未按键与已经选定发送值。
- 立即发送状态：按键后请求尽快发布一次，避免必须先等待完整周期。

每个 `PRESSED` 事件仍进入现有 `switch`。K1-K4 对应的 `case` 在执行原驾驶模式
切换前设置 `payload[0]`，启用周期发送，并标记立即发送。载荷其余 31 字节、目标
地址和 `request_id` 结构保持不变。

主循环在按键处理后统一检查发送条件：发送已启用、NRF24L01 处于 `RECEIVE`，且
存在立即发送请求或距离上次成功发布已满 1000 ms。满足条件时递增
`request_id`，发布 `nrf24l01_tx_request`，更新最后发送时间并清除立即发送状态。

## 繁忙处理

NRF24L01 的发送 Topic 采用单在途语义，`TRANSMITTING` 或 `RECOVERING` 状态下
直接发布可能丢失请求。因此繁忙期间不发布，立即发送状态保留到模块重新进入
`RECEIVE`。

繁忙期间若连续按下多个键，不建立历史按键队列；最后一次按键覆盖
`payload[0]`，恢复后只发送最新值。这与“最后一次按键选择持续发送值”的语义
一致。

## 现有行为兼容

- K1 继续进入编码器闭环定距前进模式。
- K2 继续进入循迹模式。
- K3 继续退出运动模式并滑行停止。
- K4 继续切换原地旋转与停止。
- K4 再次按下并停止运动后，发送值仍保持为 `4`，因为发送状态由最后一次按键
  决定，而不是由 `DriveMode` 推导。
- 删除当前上电后无条件执行的每秒周期发送路径。

## 验证

更新现有 PowerShell 结构检查，验证：

- K1-K4 的四个 `case` 分别写入 `payload[0] = 1/2/3/4`。
- 未按键时发送门控保持关闭。
- 按键后存在立即发送路径，后续发送仍受 1000 ms 周期约束。
- 仅在 `NRF24L01::State::RECEIVE` 时发布。
- 每次发布前递增 `request_id`，发布后更新时间并清除立即发送状态。
- 原有四个按键驾驶控制分支仍然存在。
