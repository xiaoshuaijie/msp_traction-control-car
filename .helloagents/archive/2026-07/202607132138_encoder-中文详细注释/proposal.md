## 1. 需求

### 背景
`module/encoder` 的当前实现包含正交解码、中断临界区、有符号模计数、角度/角速度换算及 Topic 周期发布等逻辑，但现有英文简注无法充分说明嵌入式并发与溢出语义。

### 目标
- 为 `encoder.hpp`、`encoder.cpp`、`encoder_math.hpp` 和 `CMakeLists.txt` 添加详细中文注释。
- 让公共接口、数据单位、生命周期、中断安全和计算边界可在不阅读全部实现的情况下理解。
- 严格保持现有程序行为、接口、构建关系和代码格式不变。

### 约束条件
```yaml
时间约束: 无
性能约束: 注释不进入固件产物，不得引入运行时开销
兼容性约束: 不修改任何非注释 token、公共 API、宏或 CMake 命令
业务约束: 以当前工作树代码为唯一事实来源，不恢复旧实现
```

### 验收标准
- [x] 4 个目标文件均包含与当前实现一致的中文注释。
- [x] 公共类型和方法使用 Doxygen 风格，关键实现块使用行内注释，不逐行复述显而易见的语句。
- [x] 非注释 token 与修改前一致，Encoder 相关逻辑测试、结构断言和 ARM 构建检查通过。

## 2. 方案

### 技术方案
- `encoder.hpp`：说明类职责、构造参数、禁止拷贝原因、状态转换表、配置项、采样字段单位和成员状态。
- `encoder.cpp`：说明 RAII 中断保护、GPIO 初始化顺序、模 `2^32` 计数、四轮快照、采样节流和 Topic 发布顺序。
- `encoder_math.hpp`：说明输入/输出单位、无效参数处理、首帧零速度和跨有符号边界的差分恢复。
- `CMakeLists.txt`：说明头文件搜索路径和 `encoder.cpp` 加入 `xr` 目标的作用。

### 影响范围
```yaml
涉及模块:
  - Encoder: 仅提升源码可读性和维护性
预计变更文件: 4 个代码/构建文件 + 1 个注释契约测试 + 最小知识库同步
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 注释与实现语义不一致 | 中 | 逐个对照状态转移、时间和溢出计算，以代码为准 |
| 添加注释时误改程序 token | 中 | 查看变更并运行去注释前后对比 |
| 中文编码问题 | 低 | 使用 UTF-8 无 BOM 写入并执行编译检查 |

## 3. 技术设计

### 注释层次
- 文件/类级：解释模块边界、调用方式与生命周期。
- 接口级：解释参数、返回值、单位、前置条件和副作用。
- 算法级：解释“为什么这样做”，包括临界区、无符号环绕和采样状态。
- 显而易见的赋值、循环和命名不做逐句翻译式注释。
